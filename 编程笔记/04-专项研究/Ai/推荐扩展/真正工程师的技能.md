# 真正工程师的技能

> [!quote] 定位
> 这些是我每天用来做**真正工程**的智能体技能——而非氛围编码(vibe coding)。
>
> —— Matt Pocock

这是 Matt Pocock（TypeScript 专家）开源的一套 Claude Code [[skills|技能]] 集合。与 GSD、BMAD、Spec-Kit 等「接管整个开发流程」的方案不同，它的设计目标是**小巧、易改造、可组合**——不夺走你的控制权，而是把数十年工程经验凝练成可插拔的纪律。

> [!info] 它解决的四大失败模式
> AI 编码智能体最常在四个地方翻车，这套技能逐一对应：
>
> | 失败模式 | 根因 | 对应技能 |
> | --- | --- | --- |
> | **错位**——智能体没做你想要的 | 你与智能体间有沟通鸿沟 | `/grill-me`、`/grill-with-docs` |
> | **啰嗦**——用 20 个词说清能 1 个词说的事 | 缺乏共享领域语言 | `/grill-with-docs`（建 `CONTEXT.md`） |
> | **跑不通**——产出垃圾代码 | 缺乏反馈环 | `/tdd`、`/diagnosing-bugs` |
> | **泥球**——代码库快速腐化 | 加速编码也加速了软件熵增 | `/improve-codebase-architecture` |

## 怎么安装

通过 skills.sh 安装器，30 秒完成：

```bash
npx skills@latest add mattpocock/skills
```

安装器会让你挑选想要的技能，以及要把它们安装到**哪些编码智能体**上（Claude Code、Codex 等多宿主支持）。

技能在安装器里分两大组勾选：

- **Mattpocock Skills**（17 个）：稳定核心，日常首选——**绝大多数人勾这组就够**。
- **Other**（18 个）：含实验性（`in-progress`）、个人用（`personal`）、杂项（`misc`）与已弃用（`deprecated`）技能，按需勾选。

> [!tip] 别勾 deprecated
> Other 组里 `design-an-interface`、`qa`、`request-refactor-plan`、`ubiquitous-language` 四个已弃用（功能已合并进核心或不再维护），无需安装。完整清单与状态见下方「技能速查」。

> [!warning] 务必勾选 /setup-matt-pocock-skills
> 在安装器里选择技能时，**一定要勾选 `/setup-matt-pocock-skills`**。它负责为你的仓库做一次性配置（issue 跟踪器、分流标签、领域文档存放位置），是其他工程技能正常工作的前提。

安装完成后，在智能体中运行该配置技能：

```
/setup-matt-pocock-skills
```

它会依次询问：

1. **用哪个 issue 跟踪器**——GitHub、Linear，或本地文件；
2. **分流工单时打哪些标签**（供 `/triage` 使用）；
3. **创建的文档保存到哪里**。

配置完即可使用。

## 怎么配

这套技能**不需要传统意义上的配置文件**——它的「配置」就是 `/setup-matt-pocock-skills` 跑的那几个问题。理解它的组织方式比记配置更重要：

### 技能的两条调用轴

所有技能沿**谁能调用它**划分为两类，这是理解整套技能的关键：

| 类型 | 触发方式 | 职责 | 例子 |
| --- | --- | --- | --- |
| **用户调用（user-invoked）** | 仅当你输入 `/xxx` 时触发 | **编排**——组织流程 | `/grill-me`、`/to-prd` |
| **模型调用（model-invoked）** | 你调用**或**智能体在任务契合时自动调用 | 承载**可复用的纪律** | `/tdd`、`/grilling` |

> [!important] 调用规则
> 用户调用的技能可以调用模型调用的技能，但**绝不能调用另一个用户调用的技能**。这条规则防止编排逻辑互相嵌套失控。

### 技能的分组与状态

安装器（`npx skills@latest add mattpocock/skills`）把 35 个技能分成**两大组**供你勾选，背后对应仓库的 6 个目录：

| 安装器分组 | 定位 | 对应仓库目录 | 数量 |
| --- | --- | --- | --- |
| **Mattpocock Skills** | 稳定的核心技能，日常首选 | `engineering` + `productivity` | 17 |
| **Other** | 实验性、个人用、已弃用的技能 | `in-progress` + `personal` + `misc` + `deprecated` | 18 |

> [!tip] 选哪些装
> - **绝大多数人只需勾 Mattpocock Skills 这一组**——它是稳定可依赖的核心。
> - **Other 组里关注 `in-progress` 的实验技能**（如 `review`、`loop-me`）——它们有潜力但尚未稳定。
> - **`deprecated` 目录的技能已弃用**（`design-an-interface`、`qa`、`request-refactor-plan`、`ubiquitous-language`），功能已合并进核心技能或不再维护，**不要装**。

完整技能清单与状态见下方「技能速查」。

## 怎么用

### 核心工作流：盘问 → 共享语言 → 反馈环 → 设计

这套技能的精髓是一条贯穿开发全过程的工作流，对应四大失败模式的解法：

#### ① 盘问对齐（动手前消除错位）

> [!quote]
> 「没有人确切知道自己想要什么。」——《程序员修炼之道》

在写任何代码前，让智能体**盘问**你，把需求里模糊的地方逼问清楚。这是最受欢迎的技能：

| 技能 | 场景 |
| --- | --- |
| `/grill-me` | 非代码场景的计划/设计盘问 |
| `/grill-with-docs` | 代码场景——盘问 + 构建领域模型 + 更新 `CONTEXT.md` 与 ADR（**最酷的单一技巧**） |

#### ② 共享语言（消除啰嗦）

建立一份 `CONTEXT.md` 文档，沉淀项目的领域术语。效果示例：

- 修改前：「当课程某章节里的一节课被『做实』（分配到文件系统位置）时出问题」
- 修改后：「**物化级联(materialization cascade)** 存在问题」

共享语言一次建立、持续受益：命名一致、代码库易导航、智能体消耗更少 token。

#### ③ 反馈环（让代码跑得通）

> [!quote]
> 「反馈的速度就是你的速度上限。绝不接下过大的任务。」——《程序员修炼之道》`

没有反馈，智能体如同盲飞。需要静态类型 + 浏览器访问 + **自动化测试**：

| 技能 | 作用 |
| --- | --- |
| `/tdd` | 红-绿-重构循环的 TDD，每次构建一个垂直切片 |
| `/diagnosing-bugs` | 严谨调试循环：复现 → 最小化 → 假设 → 插桩 → 修复 → 回归测试 |

#### ④ 架构投资（抵抗泥球）

> [!quote]
> 「每天都要为系统的设计投资。」——Kent Beck

智能体能极大地加速编码，也加速了软件熵增。解法是关心设计——建**深模块**（大量行为藏在小接口背后）：

| 技能 | 作用 |
| --- | --- |
| `/to-prd` | 把对话综合成 PRD（**不做访谈、纯综合**），梳理出要触及哪些模块 |
| `/improve-codebase-architecture` | 扫描代码库寻找「加深」机会，可视化 HTML 报告 + 盘问改进 |

## 最佳实践

> [!tip] 核心心法
> 软件工程的基本功比以往任何时候都更重要——AI 加速了编码，却没有改变什么才是好代码。这套技能的价值在于把**基本功**（澄清需求、建领域语言、TDD、关心设计）固化成可复用的纪律，而非发明新流程。

| 实践 | 说明 |
| --- | --- |
| **每次改动都先盘问** | `/grill-me` 或 `/grill-with-docs` 是最受欢迎的技能，动手前务必用——错位是最高频的失败 |
| **尽早建立 CONTEXT.md** | 共享语言一次建立、持续受益（命名一致、token 更省、代码库更易导航）。`/grill-with-docs` 会顺手帮你建 |
| **坚持红-绿-重构** | 让智能体先写失败测试再让它通过，给它稳定的反馈水平——产出质量天差地别 |
| **每几天跑一次架构扫描** | `/improve-codebase-architecture` 抵抗泥球化，别等代码库烂透了才动 |
| **用垂直切片拆工单** | `/to-issues` 拆出可独立认领、可独立交付的切片，而非大块依赖 |
| **善用 `/ask-matt` 路由** | 不确定用哪个技能时先问它，它会把流程路由到合适的位置 |
| **把它当起点而非圣旨** | 设计目标就是「易改造」——按自己的工程习惯改造成自己的东西 |

> [!example] 与「接管全流程」方案的区别
> GSD、BMAD、Spec-Kit 试图接管整个流程来帮忙，代价是夺走控制权、错误难排查。这套技能反其道——每个技能小巧独立、可组合，**你始终掌控流程**，技能只是把最佳实践喂给智能体。如果你厌倦了被工作流框架绑架，这是更对路的选择。
