# 上下文管理

## 是什么

**上下文窗口（context window）** 是 Claude 在一次会话中的「工作记忆」——它能同时「看见」的全部内容，包括系统提示、`CLAUDE.md`、对话历史、工具输出、加载的文件等。它有**硬上限**（以 token 计），超出后早期的内容会被处理掉。

Claude Code 对上下文有一套自动管理机制，核心是**压缩（compaction）**：当占用接近上限时，自动把较早的对话摘要化，腾出空间让会话继续，而不是直接中断。

> [!note] 上下文 ≠ 持久记忆
> 上下文窗口是「短期工作记忆」，会随压缩而丢失细节。需要长期保留的结论、决策、代码状态，应写进文件（`CLAUDE.md`、笔记、代码）而非指望上下文记住。社区共识：「别把上下文当主存，用文件」。

## 在哪配

上下文管理**不是某个配置文件**，而是运行时机制 + 一组斜杠命令。与之相关的设置分散在几处：

| 关注点 | 位置 / 方式 |
| --- | --- |
| 查看当前占用 | `/context` 命令（可视化） |
| 自动压缩开关 | 默认开启，遵循「最简做法」原则 |
| 模型选择（影响窗口大小） | `/model` 命令 |
| 工作目录范围 | `/add-dir` 命令扩展 |

## 怎么配

主要通过斜杠命令管理（详见 [[斜杠命令]]）：

| 命令 | 作用 | 适用时机 |
| --- | --- | --- |
| `/context [all]` | 用彩色网格可视化上下文占用，给出优化建议 | 想知道「空间都被谁占了」 |
| `/compact [instructions]` | 手动压缩对话历史，释放上下文 | 主动瘦身，可带聚焦指令保留重点 |
| `/clear [name]` | 清空上下文，开启全新对话（旧会话留在 `/resume`） | 切换到无关的新任务 |
| `/add-dir <path>` | 为会话添加额外工作目录 | 需要读取项目外的文件 |

### 压缩的工作方式

无论自动还是手动，压缩都做两件事：

1. **优先清理**较早的工具输出（如大段 grep / 文件读取结果）——这些体积大、信息密度低。
2. 若仍不够，**摘要化**对话历史，保留关键结论、丢弃冗余过程。

`/compact` 后可跟**聚焦指令**，告诉 Claude 压缩时优先保留什么：

```
/compact 保留数据库迁移相关的决策和当前错误堆栈
```

## 怎么用

### 最佳实践

> [!tip] 主动压缩优于被动压缩
> 经验法则是**在 60-70% 占用时主动 `/compact`**，而不是等到 90%+ 被动触发。被动压缩时 Claude 已处于「上下文拥挤」状态，判断力下降；主动压缩则能在状态尚好时整理思路。

| 做法 | 说明 |
| --- | --- |
| **主动 `/compact` 并带聚焦指令** | 告诉它「保留什么」，避免重要决策被摘要掉 |
| **用文件而非上下文当主存** | 关键结论写进 `CLAUDE.md` / 笔记，不依赖上下文记住 |
| **用 [[SubAgent\|子代理]] 隔离大任务** | 探索性工作丢给子代理，只把最终结果回传，不污染主上下文 |
| **不相关任务用 `/clear`** | 切换到全新主题时清空，比压缩更彻底 |
| **用 `/context` 诊断** | 占用异常时先看「谁在吃空间」 |

### 上下文管理与检查点的关系

压缩是「向前走」（整理历史、继续工作），[[Checkpoints 与 Rewind]] 是「向后退」（回滚到某个历史快照）。两者互补：压缩丢了细节但能继续，rewind 能恢复但会丢弃之后的工作。

### 避免上下文膨胀的根本手段

与其反复压缩，不如从源头减少占用：

- `CLAUDE.md` 保持精简分层（根目录放概览，子目录放局部约定）——参见 [[ClaudeCode在大型代码库中的工作方式：最佳实践与入门]]。
- 用 [[skills]] 把专门知识按需加载，而不是全塞进 `CLAUDE.md`。
- 用 `.gitignore` / `permissions.deny` 排除生成文件、构建产物，避免 Claude 误读。
