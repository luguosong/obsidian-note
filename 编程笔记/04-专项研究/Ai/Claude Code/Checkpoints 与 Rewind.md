# Checkpoints 与 Rewind

## 是什么

**检查点（checkpoint）** 是 Claude Code 在你**每次提交提示**时自动创建的快照，同时记录两样东西：

- **代码状态**：此刻工作区的文件内容（基于 git 机制实现）。
- **对话状态**：此刻的对话历史。

**Rewind（回滚）** 则是把会话「倒带」回某个检查点——既能撤销 Claude 刚写的代码，也能撤销那段跑偏的对话。它是一把**手术刀式**的恢复工具，专门对付「Claude 把方向带歪了 / 改坏了」这类情况。

> [!note] 与上下文管理的区别
> [[上下文管理]] 里的压缩是「向前走」（整理历史、腾空间继续干）；Rewind 是「向后退」（回到某个历史点重来）。压缩不可逆地丢失细节，Rewind 则能精准恢复到某个节点。

## 在哪配

**无需配置，自动启用。** 只要工作区在 git 仓库内，检查点机制就会在每次提交提示时自动生效，不需要写任何配置文件。

启用前提：

| 条件 | 说明 |
| --- | --- |
| 工作区是 git 仓库 | 检查点的代码快照依赖 git 机制实现 |
| 非交互模式（`claude -p`）下不可用 | Rewind 需要交互式会话 |

> [!warning] 非 git 环境下不可用
> 若工作区不在 git 仓库内，代码快照无法生成，Rewind 的「恢复代码」选项将不可用。

## 怎么配

无需手动配置，但了解以下开关有助于控制行为：

- 默认行为：每次提交提示自动创建检查点（对话 + 代码双快照）。
- 如需禁用自动检查点，可通过相应设置项关闭（具体环境变量 / 设置项以官方文档为准）。

> [!tip] 配合手动 git 提交
> 自动检查点是细粒度的「每提示一快照」，但它**不能完全替代有意义的 git 提交**。重大节点（完成一个功能、通过测试）仍应手动 `git commit`，形成清晰的提交历史；Rewind 用于「两次提交之间的细粒度回退」。

## 怎么用

### 触发 Rewind

两种方式等价：

| 方式 | 操作 |
| --- | --- |
| 斜杠命令 | 输入 `/rewind`（别名 `/checkpoint`、`/undo`） |
| 快捷键 | **按 `Esc` 两次** |

触发后会弹出检查点列表，按提示时间倒序排列。

### 选择恢复方式

选中某个检查点后，可选择两种恢复范围：

| 恢复选项 | 效果 | 适用场景 |
| --- | --- | --- |
| **恢复代码和对话** | 把代码与对话**都**回退到该点 | Claude 改坏了代码、对话也跑偏——彻底重来 |
| **仅恢复对话** | 只回退对话，**保留**之后的代码改动 | 代码改动是有用的，只想换个对话方向 |

> [!example] 典型场景
> 你让 Claude 重构某模块，它改了 10 个文件后方向错了。
> 1. 按 `Esc` 两次 → 弹出检查点列表；
> 2. 选中「重构前」那个检查点；
> 3. 选「恢复代码和对话」→ 代码和对话都回到重构前，从头再来。

### 最佳实践

| 做法 | 说明 |
| --- | --- |
| **早期、频繁使用** | 发现方向不对立刻 Rewind，比让 Claude 在错误基础上继续「补救」更省时 |
| **选对恢复范围** | 代码有用就「仅恢复对话」，全错就「恢复代码和对话」 |
| **重大节点手动 commit** | Rewind 之外，用 git 提交锁定确定性成果 |
| **兜底：`git reflog`** | 万一 Rewind 也救不回（极端情况），`git reflog` 能找到被丢弃的提交 |
