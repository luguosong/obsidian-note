## 通配符的更多趣味

在本节中，我们将考虑通配符的一些更高级用法。我们已经看到了几个有界通配符在从数据结构读取时很有用的示例。现在考虑相反的情况——一个只写的数据结构。接口 `Sink` 是此类的一个简单示例。

```java
interface Sink<T> {
    flush(T t);
}
```

我们可以想象按以下代码演示使用它。方法 `writeAll()` 设计为将集合 `coll` 的所有元素刷新到接收器 `snk`，并返回最后刷新的元素。

```java
public static <T> T writeAll(Collection<T> coll, Sink<T> snk) {
    T last;
    for (T t : coll) {
        last = t;
        snk.flush(last);
    }
    return last;
}
...
Sink<Object> s;
Collection<String> cs;
String str = writeAll(cs, s); // 非法调用。
```

如所写，对 `writeAll()` 的调用是非法的，因为无法推断出有效的类型参数；`String` 和 `Object` 都不是 `T` 的合适类型，因为 `Collection` 元素和 `Sink` 元素必须是相同类型。

我们可以通过如下修改 `writeAll()` 的签名来修复此错误，使用通配符。

```java
public static <T> T writeAll(Collection<? extends T>, Sink<T>) {...}
...
// 调用正常，但返回类型错误。
String str = writeAll(cs, s);
```

调用现在是合法的，但赋值是错误的，因为推断的返回类型是 `Object`，因为 `T` 匹配 `s` 的元素类型，即 `Object`。

解决方案是使用我们尚未见过的有界通配符形式：具有*下界*的通配符。语法 `? **super** T` 表示未知类型是 `T` 的超类型（或 `T` 本身；记住超类型关系是自反的）。它是我们一直使用的有界通配符的对偶，我们使用 `? **extends** T` 来表示未知类型是 `T` 的子类型。

```java
public static <T> T writeAll(Collection<T> coll, Sink<? super T> snk) {
    ...
}
String str = writeAll(cs, s); // 是的！
```

使用此语法，调用是合法的，并且推断的类型是所需的 `String`。

现在让我们转向一个更现实的示例。`java.util.TreeSet<E>` 表示有序的 `E` 类型元素的树。构造 `TreeSet` 的一种方法是将 `Comparator` 对象传递给构造器。该比较器将用于根据所需排序对 `TreeSet` 的元素进行排序。

```java
TreeSet(Comparator<E> c)
```

`Comparator` 接口本质上是：

```java
interface Comparator<T> {
    int compare(T fst, T snd);
}
```

假设我们想创建一个 `TreeSet<String>` 并传入一个合适的比较器，我们需要传递一个可以比较 `String` 的 `Comparator`。这可以通过 `Comparator<String>` 完成，但 `Comparator<Object>` 同样可行。然而，我们将无法对 `Comparator<Object>` 调用上面给出的构造器。我们可以使用下界通配符来获得我们想要的灵活性：

```java
TreeSet(Comparator<? super E> c)
```

此代码允许使用任何适用的比较器。

作为使用下界通配符的最后一个示例，让我们看看方法 `Collections.max()`，它返回作为参数传递给它的集合中的最大元素。现在，要让 `max()` 工作，传入集合的所有元素都必须实现 `Comparable`。此外，它们都必须彼此可比较。

对此方法签名进行泛型化的第一次尝试产生：

```java
public static <T extends Comparable<T>> T max(Collection<T> coll)
```

即，该方法接受某种类型 `T` 的集合（该类型与自身可比较），并返回该类型的元素。然而，此代码太受限了。要了解原因，考虑一个与任意对象可比较的类型：

```java
class Foo implements Comparable<Object> {
    ...
}
Collection<Foo> cf = ... ;
Collections.max(cf); // 应该可行。
```

`cf` 的每个元素都与 `cf` 中的每个其他元素可比较，因为每个这样的元素都是 `Foo`，它与任何对象可比较，特别是与另一个 `Foo` 可比较。然而，使用上面的签名，我们发现调用被拒绝。推断的类型必须是 `Foo`，但 `Foo` 不实现 `Comparable<Foo>`。

`T` **恰好**与自身可比较并不是必要的。所需的只是 `T` 与其超类型之一可比较。这给我们：

```java
public static <T extends Comparable<? super T>> 
        T max(Collection<T> coll)
```

注意，`Collections.max()` 的实际签名更复杂。我们将在下一节[[泛型进阶-转换遗留代码|将遗留代码转换为使用泛型]]中回到它。此推理适用于几乎所有旨在为任意类型工作的 `Comparable` 使用：你总是想使用 `Comparable<? **super** T>`。

通常，如果你有一个仅将类型参数 `T` 用作参数的 API，其使用应利用下界通配符（`? **super** T`）。相反，如果 API 仅返回 `T`，通过使用上界通配符（`? **extends** T`）将给客户端更多灵活性。

## 通配符捕获

现在应该很清楚，给定：

```java
Set<?> unknownSet = new HashSet<String>();
...
/* 向 Set s 添加元素 t。 */ 
public static <T> void addToSet(Set<T> s, T t) {
    ...
}
```

以下调用是非法的。

```java
addToSet(unknownSet, "abc"); // 非法。
```

传递的实际集合是字符串集合这一事实没有区别；重要的是作为参数传递的表达式是未知类型的集合，不能保证是字符串集合或任何特定类型的集合。

现在，考虑以下代码：

```java
class Collections {
    ...
    <T> public static Set<T> unmodifiableSet(Set<T> set) {
        ...
    }
}
...
Set<?> s = Collections.unmodifiableSet(unknownSet); // 这可行！为什么？
```

这似乎不应该被允许；然而，查看这个特定调用，允许它是完全安全的。毕竟，`unmodifiableSet()` 确实适用于任何类型的 `Set`，无论其元素类型如何。

因为这种情况出现得相对频繁，所以有一个特殊规则允许在可以证明代码安全的非常特定情况下使用此类代码。此规则称为*通配符捕获(wildcard capture)*，允许编译器将通配符的未知类型推断为泛型方法的类型参数。
