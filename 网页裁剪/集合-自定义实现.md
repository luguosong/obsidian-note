## 课程：自定义集合实现

许多程序员永远不需要实现自己的 `Collection` 类。使用本章前面各节描述的实现，你可以走得很远。然而，有一天你可能想编写自己的实现。借助 Java 平台提供的抽象实现，这相当容易。在我们讨论*如何*编写实现之前，让我们讨论为什么你可能想编写一个。

## 编写实现的原因

以下列表说明了你可能想要实现的自定义 `Collection` 类型。它并非详尽无遗：

- **持久化**：所有内置 `Collection` 实现都驻留在主内存中，并在程序退出时消失。如果你想要一个在程序下次启动时仍然存在的集合，你可以通过在外部数据库上构建外壳来实现它。这样的集合可能可以被多个程序并发访问。
- **应用程序专用**：这是一个非常广泛的类别。一个示例是包含实时遥测数据的不可修改 `Map`。键可以表示位置，值可以从这些位置的传感器读取以响应 `get` 操作。
- **高性能、专用**：许多数据结构利用受限的使用来提供比通用实现更好的性能。例如，考虑一个包含相同元素值长连续运行的 `List`。此类列表在文本处理中频繁出现，可以进行*游程编码(run-length encoded)*——运行可以表示为包含重复元素和连续重复次数的单个对象。这个示例很有趣，因为它权衡了性能的两个方面：它需要比 `ArrayList` 更少的空间但更多的时间。
- **高性能、通用**：Java 集合框架的设计者试图为每个接口提供最佳的通用实现，但可以使用许多许多数据结构，而且每天都有新的被发明。也许你能想出更快的！
- **增强功能**：假设你需要一个高效的 bag 实现（也称为*multiset*）：一个提供常量时间包含检查同时允许重复元素的 `Collection`。在 `HashMap` 之上实现这样的集合相当简单。
- **便利**：你可能想要提供超出 Java 平台所提供便利的附加实现。例如，你可能经常需要表示连续 `Integer` 范围的 `List` 实例。
- **适配器**：假设你正在使用具有自己临时集合 API 的遗留 API。你可以编写一个适配器实现，允许这些集合在 Java 集合框架中操作。*适配器实现*是一个薄外壳，它包装一种类型的对象并通过将后一种类型的操作转换为前一种类型的操作，使它们表现得像另一种类型的对象。

## 如何编写自定义实现

编写自定义实现出奇地容易。Java 集合框架提供了专门为促进自定义实现而设计的抽象实现。我们将从以下 [`Arrays.asList`](https://docs.oracle.com/javase/8/docs/api/java/util/Arrays.html#asList-T...-) 实现示例开始。

```java
public static <T> List<T> asList(T[] a) {
    return new MyArrayList<T>(a);
}

private static class MyArrayList<T> extends AbstractList<T> {

    private final T[] a;

    MyArrayList(T[] array) {
        a = array;
    }

    public T get(int index) {
        return a[index];
    }

    public T set(int index, T element) {
        T oldValue = a[index];
        a[index] = element;
        return oldValue;
    }

    public int size() {
        return a.length;
    }
}
```

信不信由你，这与 `java.util.Arrays` 中包含的实现非常接近。就这么简单！你提供构造器以及 `get`、`set` 和 `size` 方法，`AbstractList` 完成其余所有工作。你可以免费获得 `ListIterator`、批量操作、搜索操作、哈希码计算、比较和字符串表示。

假设你想使实现更快一点。抽象实现的 API 文档精确描述了每个方法是如何实现的，因此你将知道要覆盖哪些方法以获得所需的性能。前面实现的性能很好，但可以稍作改进。特别是，`toArray` 方法遍历 `List`，一次复制一个元素。鉴于内部表示，直接克隆数组要快得多也更合理。

```java
public Object[] toArray() {
    return (Object[]) a.clone();
}
```

通过添加此覆盖和更多类似的覆盖，此实现正是 `java.util.Arrays` 中的实现。为了充分披露，使用其他抽象实现要困难一些，因为你必须编写自己的迭代器，但仍然不太困难。

以下列表汇总了抽象实现：

- [`AbstractCollection`](https://docs.oracle.com/javase/8/docs/api/java/util/AbstractCollection.html) —— 既不是 `Set` 也不是 `List` 的 `Collection`。至少，你必须提供 `iterator` 和 `size` 方法。
- [`AbstractSet`](https://docs.oracle.com/javase/8/docs/api/java/util/AbstractSet.html) —— `Set`；使用与 `AbstractCollection` 相同。
- [`AbstractList`](https://docs.oracle.com/javase/8/docs/api/java/util/AbstractList.html) —— 由随机访问数据存储（如数组）支持的 `List`。至少，你必须提供*位置访问*方法（`get` 以及可选的 `set`、`remove` 和 `add`）和 `size` 方法。抽象类负责 `listIterator`（和 `iterator`）。
- [`AbstractSequentialList`](https://docs.oracle.com/javase/8/docs/api/java/util/AbstractSequentialList.html) —— 由顺序访问数据存储（如链表）支持的 `List`。至少，你必须提供 `listIterator` 和 `size` 方法。抽象类负责位置访问方法。（这与 `AbstractList` 相反。）
- [`AbstractQueue`](https://docs.oracle.com/javase/8/docs/api/java/util/AbstractQueue.html) —— 至少，你必须提供 `offer`、`peek`、`poll` 和 `size` 方法以及支持 `remove` 的 `iterator`。
- [`AbstractMap`](https://docs.oracle.com/javase/8/docs/api/java/util/AbstractMap.html) —— `Map`。至少你必须提供 `entrySet` 视图。这通常使用 `AbstractSet` 类实现。如果 `Map` 是可修改的，你还必须提供 `put` 方法。

编写自定义实现的过程如下：

1. 从前面的列表中选择适当的抽象实现类。
2. 为类的所有抽象方法提供实现。如果你的自定义集合是可修改的，你还必须覆盖一个或多个具体方法。抽象实现类的 API 文档会告诉你覆盖哪些方法。
3. 测试并在必要时调试实现。你现在有一个可工作的自定义集合实现。
4. 如果你关心性能，请阅读抽象实现类的 API 文档中你正在继承实现的所有方法。如果任何方法似乎太慢，覆盖它们。如果你覆盖任何方法，请确保在覆盖前后测量方法的性能。你在调整性能方面投入多少精力应该是该实现将获得多少使用以及其使用对性能有多关键的函数。（通常最好省略此步骤。）
