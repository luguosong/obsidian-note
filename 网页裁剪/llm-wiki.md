# 大语言模型(LLM) Wiki

## LLM Wiki

使用大规模语言模型(LLMs)构建个人知识库的模式。

这是一个想法文件，旨在复制粘贴到你自己的 LLM Agent（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）中。其目标是传达高层次的构想，但你的 agent 将与您合作来完善具体细节。

## 核心思想

大多数人使用大型语言模型(LLMs)处理文档的体验都像检索增强生成(RAG)：你上传一组文件，LLM 在查询时检索相关片段并生成答案。这种方式可行，但 LLM 在每次提问时都是从头重新发现知识。没有任何积累。提出一个需要综合五篇文档的细致问题，LLM 每次都必须查找并拼凑相关片段。没有任何内容被构建或沉淀。NotebookLM、ChatGPT 文件上传以及大多数 RAG 系统都是这样运作的。

这里的思路不同。LLM 不是在查询时仅从原始文档中检索，而是 **增量构建并维护一个持久的维基** ——一个结构化、相互链接的 Markdown 文件集合，位于你与原始来源之间。当你添加新来源时，LLM 不会只是为以后检索建立索引。它会阅读该来源、提取关键信息，并将其整合到现有维基中——更新实体页面、修订主题摘要、标注新数据与旧说法的冲突、增强或质疑不断演进的综合结论。知识只需编译一次，然后被 *保持最新* ，而不是在每次查询时重新推导。

这是关键区别： **维基是一个持久的、可积累的成果物。** 交叉引用已经存在。矛盾已经被标注。综合内容已经反映了你所阅读的一切。随着你添加的每一个来源和提出的每一个问题，维基会变得愈加丰富。

你几乎不自己撰写维基——由 LLM(LLM)来撰写和维护全部内容。你负责资料来源、探索以及提出恰当的问题。LLM(LLM)完成所有繁重工作——总结、交叉引用、归档和记账，这些工作使知识库随着时间真正有用。实际上，我通常在一侧打开 LLM(LLM)代理，另一侧打开 Obsidian(Obsidian)。LLM(LLM)根据我们的对话进行编辑，我实时浏览结果——跟随链接、检查图谱视图、阅读更新后的页面。Obsidian(Obsidian)是 IDE(IDE)；LLM(LLM)是程序员；维基是代码库。

这可以适用于许多不同的情境。举几个例子：

- **个人** ：追踪你自己的目标、健康、心理、自我提升——归档日记条目、文章、播客笔记，并随着时间推移构建关于自己的结构化图景。
- **研究** ：在数周或数月内深入研究一个主题——阅读论文、文章、报告，并逐步构建一个随着论点演进的综合性维基。
- **阅读一本书** ：在阅读过程中将每章归档，为人物、主题、情节线以及它们之间的关联建立页面。到最后你会拥有一个丰富的伴随式维基。想想像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝维基——成千上万相互链接的页面，涵盖人物、地点、事件、语言，由志愿者社区多年构建。你可以在个人阅读时构建类似的东西，由 LLM(LLM)负责所有交叉引用和维护。
- **业务/团队** ：由 LLM 维护的内部维基，由 Slack 线程、会议记录、项目文档、客户通话等提供内容输入。可能有人类在环审核更新。维基保持最新，因为 LLM 承担了团队中无人愿意做的维护工作。
- **竞争分析、尽职调查、行程规划、课程笔记、兴趣深度研究** —— 任何你随着时间积累知识并希望将其组织起来而不是四散分布的内容。

## 架构

有三层：

**原始来源** — 您精心整理的来源文档集合。文章、论文、图像、数据文件。这些是不可更改的 — LLM（大型语言模型）从中读取但绝不修改它们。它们是您的真实来源。

**维基** — 一个由 LLM 生成的 Markdown 文件目录。包括摘要、实体页面、概念页面、比较、概览与综合。这个层完全由 LLM 管理。它创建页面，在有新来源到来时更新页面，维护交叉引用，并保持所有内容的一致性。你阅读它；LLM 撰写它。

**模式** — 一份文档（例如用于 Claude Code 的 CLAUDE.md 或用于 Codex 的 AGENTS.md），指示 LLM 维基的结构、约定以及在摄取来源、回答问题或维护维基时应遵循的工作流程。这是关键的配置文件 —— 它使 LLM 成为一个有纪律的维基维护者，而不是一个通用聊天机器人。你和 LLM 会随着时间共同演化这份文件，以找出对你的领域有效的做法。

## 操作

**导入。** 你将一个新来源放入原始集合并告诉 LLM 去处理它。一个示例流程：LLM 阅读该来源，与您讨论关键要点，在维基中撰写摘要页面，更新索引，更新整个维基中相关的实体和概念页面，并在日志中追加一条条目。单个来源可能会涉及 10–15 个维基页面。我个人更倾向于一次导入一个来源并保持参与——我会阅读摘要、检查更新，并指导 LLM 应强调的内容。但你也可以批量导入许多来源并减少监督。由你来制定适合自己风格的工作流程，并将其记录在模式(schema)中以供未来会话使用。

**查询(Query).** 你针对维基提问。LLM 会搜索相关页面、阅读它们，并合成带有引用的答案。答案可根据问题采取不同形式——一页 Markdown 页面、一个比较表、一个幻灯片组（Marp）、一张图表（matplotlib）、一个画布。重要见解： **好的答案可以作为新页面归档回维基。** 你请求的比较、一次分析、你发现的关联——这些都是有价值的，不应消失在聊天记录中。通过这种方式，你的探索会像已摄取的来源一样在知识库中复利增长。

**校验(Lint).** 定期请 LLM 对维基进行健康检查。检查项包括：页面之间的矛盾、已被更新来源取代的陈旧论断、没有入链的孤立页面、被提及但缺乏独立页面的重要概念、缺失的交叉引用、可通过网络检索填补的数据空白。LLM 擅长提出新的调查问题和建议新的信息来源。这能在维基增长的同时保持其健康性。

## 索引与记录(Indexing and logging)

两个特殊文件帮助 LLM（以及你）在知识库增长时进行导航。它们具有不同的用途：

**index.md** 是以内容为导向的。它是知识库中所有内容的目录——每个页面都以链接、一行摘要列出，并可选地包含诸如日期或来源数量等元数据。按类别组织（实体、概念、来源等）。LLM 在每次摄取时更新它。回答查询时，LLM 首先读取索引以查找相关页面，然后深入这些页面。在中等规模（约 100 个来源、数百(~hundreds of pages)页面）下这出乎意料地奏效，并且避免了基于 embedding 的 RAG 基础设施的需求。

**log.md** 是按时间顺序的。它是对发生事件及其时间的追加式记录——摄取、查询、lint 运行等。有用的提示：如果每条记录以一致的前缀开头（例如 `## [2026-04-02] ingest | Article Title` ），则日志可以用简单的 unix 工具解析—— `grep "^## \[" log.md | tail -5` 会给出最近 5 条记录。日志为知识库的发展提供时间线，并帮助 LLM 理解近期已完成的工作。

## 可选：CLI 工具

到某个阶段，你可能会想构建一些小工具，帮助 LLM 更高效地在 wiki 上操作。对 wiki 页面进行搜索的引擎是最明显的一个——在小规模时，索引文件就足够，但随着 wiki 增长，你需要真正的搜索功能。 [qmd](https://github.com/tobi/qmd) 是一个不错的选择：它是针对 markdown 文件的本地搜索引擎，支持混合 BM25/向量搜索并由 LLM 进行重新排序，所有操作均在本地设备上完成。它既有 CLI（因此 LLM 可以通过 shell 调用它），也有 MCP 服务器（因此 LLM 可以将其作为本地工具使用）。你也可以自己构建更简单的东西——当需要时，LLM 可以帮助你“vibe-code”出一个简单的搜索脚本。

## 技巧与窍门

- **Obsidian Web Clipper** 是将网页文章转换为 markdown 的浏览器扩展。对于将来源快速导入原始文档集合非常有用。
- **将图片下载到本地。** 在 Obsidian 设置 → 文件与链接 中，将“附件文件夹路径”设置为固定目录（例如 `raw/assets/` ）。然后在 设置 → 快捷键 中，搜索“Download”以找到“为当前文件下载附件”，并将其绑定到快捷键（例如 Ctrl+Shift+D）。在剪辑文章后，按下该快捷键，所有图片将下载到本地磁盘。此操作为可选但有用——它使 LLM（大语言模型）能够直接查看并引用图片，而不是依赖可能失效的 URL。注意，LLM 无法在一次处理过程中原生读取包含内联图片的 markdown——一种变通方法是先让 LLM 阅读文本，然后单独查看部分或全部被引用的图片以获取额外上下文。虽然有些笨拙，但效果足够好。
- **Obsidian 的图谱视图** 是查看你的维基结构的最佳方式——哪些内容相互连接、哪些页面是枢纽、哪些是孤立节点。
- **Marp** 是一种基于 markdown 的幻灯片格式。Obsidian 有对应的插件。对于直接从维基内容生成演示文稿非常有用。
- **Dataview** 是一个 Obsidian 插件，用于对页面的 frontmatter 运行查询。如果你的 LLM 在 wiki 页面中添加了 YAML frontmatter（标签(tags)、日期(dates)、来源计数(source counts)），Dataview 可以生成动态表格和列表。
- 这个 wiki 只是一个存放 Markdown 文件的 git 仓库。你可以免费获得版本历史、分支和协作功能。

## 为什么这能奏效

维护知识库中最繁琐的部分不是阅读或思考——而是簿记工作。更新交叉引用、保持摘要最新、记录何时新数据与旧结论相矛盾、在数十个页面间维持一致性。人类会放弃维护 wiki，因为维护负担增长得比价值快。LLM 不会感到厌倦，也不会忘记更新交叉引用，并且可以在一次操作中修改 15 个文件。由于维护成本几乎为零，wiki 因而保持被维护。

人的工作是策划资料来源、指导分析、提出好问题，并思考这一切意味着什么。LLM 的工作则是其他一切。

这一想法在精神上与 Vannevar Bush 的 Memex (1945) 相呼应——一个个人的、策划的知识存储，文档之间存在关联路径。Bush 的愿景比起网络最终的发展更接近此处：私有的、积极策划的，文档之间的连接与文档本身一样有价值。他未能解决的部分是由谁来维护。LLM 负责这项维护。

## 注

本文件故意保持抽象。它描述的是理念，而非具体实现。确切的目录结构、模式(schema)、页面格式、工具链——这些都将取决于你的领域、偏好以及所选的 LLM(LLM)。上述所有内容均为可选且模块化——选择有用的，忽略不需要的。例如：你的源材料可能仅为文本，因而完全不需要处理图像。你的维基可能足够小，索引文件就已足够，无需搜索引擎。你可能不关心幻灯片，仅需要 Markdown 页面。你也可能需要一套截然不同的输出格式。正确的使用方式是将其与您的 LLM(LLM)代理共享，并共同工作以实例化一个符合你需求的版本。本文档的唯一职责是传达这一模式。剩下的让你的 LLM(LLM)来解决。
