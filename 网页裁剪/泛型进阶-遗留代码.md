## 与遗留代码互操作

到目前为止，我们所有的示例都假设一个理想化的世界，每个人都使用支持泛型的最新版本 Java 编程语言。

唉，现实中并非如此。数百万行代码是用早期版本的语言编写的，它们不会一夜之间全部转换。

稍后，在[[泛型进阶-转换遗留代码|将遗留代码转换为使用泛型]]一节中，我们将解决将旧代码转换为使用泛型的问题。在本节中，我们将关注一个更简单的问题：遗留代码和泛型代码如何互操作？这个问题有两个部分：在泛型代码中使用遗留代码，以及在遗留代码中使用泛型代码。

## 在泛型代码中使用遗留代码

如何使用旧代码，同时仍在自己代码中享受泛型的好处？

例如，假设你想使用包 `com.Example.widgets`。Example.com 的人销售一个库存控制系统，其要点如下所示：

```java
package com.Example.widgets;

public interface Part {...}

public class Inventory {
    /**
     * 向库存数据库添加一个新的 Assembly。
     * 该 assembly 被赋予名称 name，并
     * 由 parts 指定的部件集合组成。
     * 集合 parts 的所有元素
     * 必须支持 Part 接口。
     **/ 
    public static void addAssembly(String name, Collection parts) {...}
    public static Assembly getAssembly(String name) {...}
}

public interface Assembly {
    // 返回 Part 的集合
    Collection getParts();
}
```

现在，你想添加使用上述 API 的新代码。确保你总是用正确的参数调用 `addAssembly()` 会很好——即，你传入的集合确实是 `Part` 的 `Collection`。当然，泛型正是为此量身定制的：

```java
package com.mycompany.inventory;

import com.Example.widgets.*;

public class Blade implements Part {
    ...
}

public class Guillotine implements Part {
}

public class Main {
    public static void main(String[] args) {
        Collection<Part> c = new ArrayList<Part>();
        c.add(new Guillotine()) ;
        c.add(new Blade());
        Inventory.addAssembly("thingee", c);
        Collection<Part> k = Inventory.getAssembly("thingee").getParts();
    }
}
```

当我们调用 `addAssembly` 时，它期望第二个参数的类型为 `Collection`。实际参数的类型为 `Collection<Part>`。这可行，但为什么？毕竟，大多数集合不包含 `Part` 对象，因此通常情况下，编译器无法知道类型 `Collection` 指的是什么类型的集合。

在正确的泛型代码中，`Collection` 总是伴随着类型参数。当像 `Collection` 这样的泛型类型在没有类型参数的情况下使用时，它被称为*原始类型(raw type)*。

大多数人的第一直觉是 `Collection` 真正意味着 `Collection<Object>`。然而，正如我们之前看到的，在需要 `Collection<Object>` 的地方传递 `Collection<Part>` 是不安全的。更准确地说，类型 `Collection` 表示某种未知类型的集合，就像 `Collection<?>` 一样。

但等等，这也不对！考虑对 `getParts()` 的调用，它返回 `Collection`。然后将其赋值给 `k`，它是 `Collection<Part>`。如果调用的结果是 `Collection<?>`，则赋值将是错误的。

实际上，赋值是合法的，但它会产生*未检查警告(unchecked warning)*。需要此警告，因为事实是编译器无法保证其正确性。我们无法检查 `getAssembly()` 中的遗留代码，以确保返回的集合确实是 `Part` 的集合。代码中使用的类型是 `Collection`，人们可以合法地向此类集合中插入各种对象。

那么，这不应该是个错误吗？理论上讲，是的；但实际上讲，如果泛型代码要调用遗留代码，就必须允许这样做。这取决于你（程序员）自己确信在这种情况下赋值是安全的，因为 `getAssembly()` 的契约说它返回 `Part` 的集合，即使类型签名没有显示这一点。

所以原始类型非常像通配符类型，但它们的类型检查不那么严格。这是一个有意的设计决策，目的是允许泛型与预先存在的遗留代码互操作。

从泛型代码调用遗留代码本质上是危险的；一旦你将泛型代码与非泛型遗留代码混合，泛型类型系统通常提供的所有安全保证都将失效。然而，你仍然比完全不使用泛型要好。至少你知道你这端的代码是一致的。

目前非泛型代码比泛型代码多得多，不可避免地会出现它们必须混合的情况。

如果你发现自己必须混合遗留代码和泛型代码，请密切注意未检查警告。仔细思考如何证明导致警告的代码是安全的。

如果你仍然犯了错误，导致警告的代码确实不是类型安全的，会发生什么？让我们看看这种情况。在此过程中，我们将对编译器的工作有一些了解。

## 擦除和转换

```java
public String loophole(Integer x) {
    List<String> ys = new LinkedList<String>();
    List xs = ys;
    xs.add(x); // 编译时未检查警告
    return ys.iterator().next();
}
```

这里，我们为字符串列表和普通旧列表起了别名。我们向列表中插入一个 `Integer`，并尝试提取一个 `String`。这显然是错误的。如果我们忽略警告并尝试执行此代码，它将恰好在我们尝试使用错误类型的时候失败。在运行时，此代码的行为类似于：

```java
public String loophole(Integer x) {
    List ys = new LinkedList;
    List xs = ys;
    xs.add(x); 
    return(String) ys.iterator().next(); // 运行时错误
}
```

当我们从列表中提取元素并尝试通过强制转换为 `String` 来将其视为字符串时，我们将得到一个 `ClassCastException`。泛型版本的 `loophole()` 也会发生完全相同的事情。

原因是，泛型由 Java 编译器作为名为*擦除(erasure)*的前端转换实现。你可以（几乎）将其视为源到源的转换，其中泛型版本的 `loophole()` 被转换为非泛型版本。

因此，**即使存在未检查警告，Java 虚拟机的类型安全和完整性也永远不会受到威胁**。

基本上，擦除去除（或*擦除*）所有泛型类型信息。尖括号之间的所有类型信息都被丢弃，因此，例如像 `List<String>` 这样的参数化类型被转换为 `List`。类型变量的所有剩余使用都被类型变量的上界（通常是 `Object`）替换。并且，每当生成的代码类型不正确时，就插入到适当类型的强制转换，如 `loophole` 的最后一行。

擦除的完整细节超出了本教程的范围，但我们刚才给出的简单描述与真相相差不远。了解一下这些是有好处的，特别是如果你想做一些更复杂的事情，比如将现有 API 转换为使用泛型（参见[[泛型进阶-转换遗留代码|将遗留代码转换为使用泛型]]一节），或者只是想理解事情为什么是这样。

## 在遗留代码中使用泛型代码

现在让我们考虑相反的情况。想象 Example.com 选择将其 API 转换为使用泛型，但他们的一些客户端还没有。所以现在代码看起来像：

```java
package com.Example.widgets;

public interface Part { 
    ...
}

public class Inventory {
    /**
     * 向库存数据库添加一个新的 Assembly。
     * 该 assembly 被赋予名称 name，并
     * 由 parts 指定的部件集合组成。
     * 集合 parts 的所有元素
     * 必须支持 Part 接口。
     **/ 
    public static void addAssembly(String name, Collection<Part> parts) {...}
    public static Assembly getAssembly(String name) {...}
}

public interface Assembly {
    // 返回 Part 的集合
    Collection<Part> getParts();
}
```

而客户端代码看起来像：

```java
package com.mycompany.inventory;

import com.Example.widgets.*;

public class Blade implements Part {
...
}

public class Guillotine implements Part {
}

public class Main {
    public static void main(String[] args) {
        Collection c = new ArrayList();
        c.add(new Guillotine()) ;
        c.add(new Blade());

        // 1: 未检查警告
        Inventory.addAssembly("thingee", c);

        Collection k = Inventory.getAssembly("thingee").getParts();
    }
}
```

客户端代码是在引入泛型之前编写的，但它使用包 `com.Example.widgets` 和集合库，两者都使用泛型类型。客户端代码中泛型类型声明的所有使用都是原始类型。

第 1 行产生未检查警告，因为在期望 `Part` 的 `Collection` 的地方传递了原始 `Collection`，而编译器无法确保原始 `Collection` 确实是 `Part` 的 `Collection`。

或者，你可以使用 source 1.4 标志编译客户端代码，确保不生成警告。然而，在这种情况下，你将无法使用 JDK 5.0 中引入的任何新语言特性。
