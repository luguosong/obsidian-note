## 将遗留代码转换为使用泛型

之前，我们展示了新代码和遗留代码如何互操作。现在，是时候看看"泛型化"旧代码这一更难的问题了。

如果你决定将旧代码转换为使用泛型，你需要仔细考虑如何修改 API。

你需要确保泛型 API 不会过于受限；它必须继续支持 API 的原始契约。再次考虑 `java.util.Collection` 中的一些示例。泛型前的 API 看起来像：

```java
interface Collection {
    public boolean containsAll(Collection c);
    public boolean addAll(Collection c);
}
```

泛型化它的天真尝试将是：

```java
interface Collection<E> {

    public boolean containsAll(Collection<E> c);
    public boolean addAll(Collection<E> c);
}
```

虽然这当然是类型安全的，但它不符合 API 的原始契约。`containsAll()` 方法适用于任何类型的传入集合。它只有在传入集合确实只包含 `E` 的实例时才会成功，但是：

- 传入集合的静态类型可能不同，也许是因为调用者不知道传入集合的精确类型，或者也许它是 `Collection<S>`，其中 `S` 是 `E` 的子类型。
- 用不同类型的集合调用 `containsAll()` 是完全合法的。例程应该工作，返回 `false`。

在 `addAll()` 的情况下，我们应该能够添加任何由 `E` 的子类型实例组成的集合。我们在[[泛型进阶-泛型方法|泛型方法]]一节中看到了如何正确处理这种情况。

你还需要确保修订后的 API 与旧客户端保持二进制兼容。这意味着 API 的擦除必须与原始的、未泛型化的 API 相同。在大多数情况下，这自然就出来了，但有一些微妙的情况。我们将检查我们遇到的最微妙的情况之一，方法 `Collections.max()`。正如我们在[[泛型进阶-通配符进阶|通配符的更多趣味]]一节中所见，`max()` 的一个合理签名是：

```java
public static <T extends Comparable<? super T>> 
        T max(Collection<T> coll)
```

这没问题，除了此签名的擦除是：

```java
public static Comparable max(Collection coll)
```

这与 `max()` 的原始签名不同：

```java
public static Object max(Collection coll)
```

当然可以为 `max()` 指定此签名，但并没有这样做，并且所有调用 `Collections.max()` 的旧二进制类文件都依赖于返回 `Object` 的签名。

我们可以通过为形式类型参数 `T` 的边界显式指定超类来强制擦除不同。

```java
public static <T extends Object & Comparable<? super T>> 
        T max(Collection<T> coll)
```

这是为类型参数指定*多重边界(multiple bounds)*的示例，使用语法 `T1 & T2 ... & Tn`。具有多重边界的类型变量已知是边界中列出的所有类型的子类型。当使用多重边界时，边界中提到的第一个类型用作类型变量的擦除。

最后，我们应该回想 `max` 只从其输入集合读取，因此适用于 `T` 的任何子类型的集合。

这使我们得出 JDK 中使用的实际签名：

```java
public static <T extends Object & Comparable<? super T>> 
        T max(Collection<? extends T> coll)
```

在实践中很少出现如此复杂的情况，但专家库设计者在转换现有 API 时应准备好非常仔细地思考。

另一个需要注意的问题是*协变返回(covariant returns)*，即在子类中细化方法的返回类型。你不应在旧 API 中利用此特性。要了解原因，让我们看一个示例。

假设你的原始 API 形式为：

```java
public class Foo {
    // 工厂。应创建
    // 其声明所在的类的实例。
    public Foo create() {
        ...
    }
}

public class Bar extends Foo {
    // 实际创建一个 Bar。
    public Foo create() {
        ...
    }
}
```

利用协变返回，你将其修改为：

```java
public class Foo {
    // 工厂。应创建
    // 其声明所在的类的实例。
    public Foo create() {
        ...
    }
}

public class Bar extends Foo {
    // 实际创建一个 Bar。
    public Bar create() {
        ...
    }
}
```

现在，假设你代码的第三方客户端编写了以下内容：

```java
public class Baz extends Bar {
    // 实际创建一个 Baz。
    public Foo create() {
        ...
    }
}
```

Java 虚拟机不直接支持具有不同返回类型的方法覆盖。此特性由编译器支持。因此，除非重新编译类 `Baz`，否则它不会正确覆盖 `Bar` 的 `create()` 方法。此外，`Baz` 必须修改，因为代码将按所写被拒绝——`Baz` 中 `create()` 的返回类型不是 `Bar` 中 `create()` 返回类型的子类型。
