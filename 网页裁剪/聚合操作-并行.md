## 并行

并行计算涉及将问题划分为子问题、同时解决这些问题（并行，每个子问题在单独的线程中运行），然后合并子问题的解决方案结果。Java SE 提供了[[并发-forkjoin框架|fork/join 框架]]，它使你能够更轻松地在应用程序中实现并行计算。但是，使用此框架，你必须指定如何细分（分区）问题。使用聚合操作，Java 运行时为你执行此分区和解决方案合并。

在使用集合的应用程序中实现并行的一个困难是集合不是线程安全的，这意味着多个线程无法操作集合而不引入[[并发-线程干扰|线程干扰]]或[[并发-内存一致性错误|内存一致性错误]]。集合框架提供[[集合-实现-包装器实现|同步包装器]]，它们为任意集合添加自动同步，使其线程安全。但是，同步引入[[并发-同步|线程争用]]。你希望避免线程争用，因为它阻止线程并行运行。聚合操作和并行流使你能够用非线程安全的集合实现并行，前提是你在操作时不修改集合。

注意，并行并不自动比串行执行操作更快，尽管如果你有足够的数据和处理器核心，它可以更快。虽然聚合操作使你能够更轻松地实现并行，但确定你的应用程序是否适合并行仍然是你的责任。

## 并行执行流

你可以串行或并行执行流。当流并行执行时，Java 运行时将流划分为多个子流。聚合操作并行迭代并处理这些子流，然后合并结果。

创建流时，除非另有指定，它始终是串行流。要创建并行流，调用操作 [`Collection.parallelStream`](https://docs.oracle.com/javase/8/docs/api/java/util/Collection.html#parallelStream--)。或者，调用操作 [`BaseStream.parallel`](https://docs.oracle.com/javase/8/docs/api/java/util/stream/BaseStream.html#parallel--)。例如，以下语句并行计算所有男性成员的平均年龄：

```java
double average = roster
    .parallelStream()
    .filter(p -> p.getGender() == Person.Sex.MALE)
    .mapToInt(Person::getAge)
    .average()
    .getAsDouble();
```

## 并发归约

再次考虑以下示例（在[[聚合操作-归约|归约]]节中描述），它按性别对成员进行分组。此示例调用 `collect` 操作，将集合 `roster` 归约为 `Map`：

```java
Map<Person.Sex, List<Person>> byGender =
    roster
        .stream()
        .collect(
            Collectors.groupingBy(Person::getGender));
```

以下是并行等效代码：

```java
ConcurrentMap<Person.Sex, List<Person>> byGender =
    roster
        .parallelStream()
        .collect(
            Collectors.groupingByConcurrent(Person::getGender));
```

这称为*并发归约*。如果对于包含 `collect` 操作的特定流水线，以下条件全部为真，则 Java 运行时执行并发归约：

- 流是并行的。
- `collect` 操作的参数（收集器）具有特征 [`Collector.Characteristics.CONCURRENT`](https://docs.oracle.com/javase/8/docs/api/java/util/stream/Collector.Characteristics.html#CONCURRENT)。要确定收集器的特征，调用 [`Collector.characteristics`](https://docs.oracle.com/javase/8/docs/api/java/util/stream/Collector.Characteristics.html) 方法。
- 流是无序的，或者收集器具有特征 [`Collector.Characteristics.UNORDERED`](https://docs.oracle.com/javase/8/docs/api/java/util/stream/Collector.Characteristics.html#UNORDERED)。要确保流是无序的，调用 [`BaseStream.unordered`](https://docs.oracle.com/javase/8/docs/api/java/util/stream/BaseStream.html#unordered--) 操作。

**注意**：此示例返回 [`ConcurrentMap`](https://docs.oracle.com/javase/8/docs/api/java/util/concurrent/ConcurrentMap.html) 的实例而不是 `Map`，并调用 [`groupingByConcurrent`](https://docs.oracle.com/javase/8/docs/api/java/util/stream/Collectors.html#groupingByConcurrent-java.util.function.Function-) 操作而不是 `groupingBy`。（有关 `ConcurrentMap` 的更多信息，请参阅[[并发-并发集合|并发集合]]节。）与 `groupingByConcurrent` 操作不同，`groupingBy` 操作在并行流上性能很差。（因为它通过按键合并两个映射来操作，这在计算上很昂贵。）类似地，操作 [`Collectors.toConcurrentMap`](https://docs.oracle.com/javase/8/docs/api/java/util/stream/Collectors.html#toConcurrentMap-java.util.function.Function-java.util.function.Function-) 在并行流上比操作 [`Collectors.toMap`](https://docs.oracle.com/javase/8/docs/api/java/util/stream/Collectors.html#toMap-java.util.function.Function-java.util.function.Function-) 性能更好。

## 顺序

流水线处理流元素的顺序取决于流是串行还是并行执行、流的源以及中间操作。例如，考虑以下示例，它用 `forEach` 操作多次打印 `ArrayList` 实例的元素：

```java
Integer[] intArray = {1, 2, 3, 4, 5, 6, 7, 8 };
List<Integer> listOfIntegers =
    new ArrayList<>(Arrays.asList(intArray));

System.out.println("listOfIntegers:");
listOfIntegers
    .stream()
    .forEach(e -> System.out.print(e + " "));
System.out.println("");

System.out.println("listOfIntegers sorted in reverse order:");
Comparator<Integer> normal = Integer::compare;
Comparator<Integer> reversed = normal.reversed();
Collections.sort(listOfIntegers, reversed);
listOfIntegers
    .stream()
    .forEach(e -> System.out.print(e + " "));
System.out.println("");

System.out.println("Parallel stream");
listOfIntegers
    .parallelStream()
    .forEach(e -> System.out.print(e + " "));
System.out.println("");

System.out.println("Another parallel stream:");
listOfIntegers
    .parallelStream()
    .forEach(e -> System.out.print(e + " "));
System.out.println("");

System.out.println("With forEachOrdered:");
listOfIntegers
    .parallelStream()
    .forEachOrdered(e -> System.out.print(e + " "));
System.out.println("");
```

此示例由五个流水线组成。它打印类似以下的输出：

```text
listOfIntegers:
1 2 3 4 5 6 7 8
listOfIntegers sorted in reverse order:
8 7 6 5 4 3 2 1
Parallel stream:
3 4 1 6 2 5 7 8
Another parallel stream:
6 3 1 5 7 8 4 2
With forEachOrdered:
8 7 6 5 4 3 2 1
```

此示例执行以下操作：

- 第一个流水线按元素添加到列表的顺序打印列表 `listOfIntegers` 的元素。
- 第二个流水线在通过方法 [`Collections.sort`](https://docs.oracle.com/javase/8/docs/api/java/util/Collections.html#sort-java.util.List-) 排序后打印 `listOfIntegers` 的元素。
- 第三和第四个流水线以明显随机的顺序打印列表的元素。记住，流操作在处理流的元素时使用内部迭代。因此，当你并行执行流时，Java 编译器和运行时确定处理流元素的顺序以最大化并行计算的好处，除非流操作另有指定。
- 第五个流水线使用方法 [`forEachOrdered`](https://docs.oracle.com/javase/8/docs/api/java/util/stream/Stream.html#forEachOrdered-java.util.function.Consumer-)，它按其源指定的顺序处理流的元素，无论你是串行还是并行执行流。注意，如果你将 `forEachOrdered` 等操作与并行流一起使用，你可能会失去并行的好处。

## 副作用

如果方法或表达式除了返回或产生值之外，还修改计算机的状态，则它具有副作用。示例包括可变归约（使用 `collect` 操作的操作；有关更多信息，请参阅[[聚合操作-归约|归约]]节）以及为调试调用 `System.out.println` 方法。JDK 能很好地处理流水线中的某些副作用。特别是，`collect` 方法设计为以并行安全的方式执行具有副作用的最常见流操作。像 `forEach` 和 `peek` 这样的操作是为副作用设计的；返回 void 的 lambda 表达式（例如调用 `System.out.println` 的表达式）除了产生副作用什么也做不了。即便如此，你也应谨慎使用 `forEach` 和 `peek` 操作；如果你将其中一个操作与并行流一起使用，则 Java 运行时可能从多个线程并发地调用你指定为其参数的 lambda 表达式。此外，永远不要将具有副作用的 lambda 表达式作为参数传递给 `filter` 和 `map` 等操作。以下各节讨论[干扰](#interference)和[有状态 lambda 表达式](#stateful_lambda_expressions)，两者都可能是副作用的来源并可能返回不一致或不可预测的结果，尤其是在并行流中。但是，首先讨论[惰性](#laziness)的概念，因为它对干扰有直接影响。

### 惰性

所有中间操作都是*惰性的*。如果表达式、方法或算法的值仅在需要时才计算，则它是惰性的。（如果立即计算或处理，则算法是*急切的*。）中间操作是惰性的，因为它们直到终端操作开始才开始处理流的内容。惰性处理流使 Java 编译器和运行时能够优化它们处理流的方式。例如，在 `filter` - `mapToInt` - `average` 这样的流水线中（如[[聚合操作|聚合操作]]节所述），`average` 操作可以从 `mapToInt` 操作创建的流中获取前几个整数，而 `mapToInt` 操作从 `filter` 操作获取元素。`average` 操作将重复此过程，直到它从流中获取所有所需元素，然后计算平均值。

### 干扰

流操作中的 lambda 表达式不应*干扰*。当流水线处理流时修改流的源时，就会发生干扰。例如，以下代码尝试连接 `List` `listOfStrings` 中包含的字符串。但是，它抛出 `ConcurrentModificationException`：

```java
try {
    List<String> listOfStrings =
        new ArrayList<>(Arrays.asList("one", "two"));

    // 这将失败，因为 peek 操作会在终端操作开始后
    // 尝试将字符串 "three" 添加到源中。

    String concatenatedString = listOfStrings
        .stream()

        // 不要这样做！这里会发生干扰。
        .peek(s -> listOfStrings.add("three"))

        .reduce((a, b) -> a + " " + b)
        .get();

    System.out.println("Concatenated string: " + concatenatedString);

} catch (Exception e) {
    System.out.println("Exception caught: " + e.toString());
}
```

此示例使用 `reduce` 操作将 `listOfStrings` 中包含的字符串连接为 `Optional<String>` 值，`reduce` 是终端操作。但是，此处的流水线调用中间操作 `peek`，它尝试向 `listOfStrings` 添加新元素。记住，所有中间操作都是惰性的。这意味着此示例中的流水线在调用操作 `get` 时开始执行，并在 `get` 操作完成时结束执行。`peek` 操作的参数尝试在流水线执行期间修改流源，这导致 Java 运行时抛出 `ConcurrentModificationException`。

### 有状态 lambda 表达式

避免使用*有状态 lambda 表达式*作为流操作的参数。有状态 lambda 表达式是其结果依赖于在流水线执行期间可能更改的任何状态的表达式。以下示例使用 `map` 中间操作将 `List` `listOfIntegers` 中的元素添加到新的 `List` 实例。它执行两次，首先用串行流，然后用并行流：

```java
List<Integer> serialStorage = new ArrayList<>();

System.out.println("Serial stream:");
listOfIntegers
    .stream()

    // 不要这样做！它使用了有状态 lambda 表达式。
    .map(e -> { serialStorage.add(e); return e; })

    .forEachOrdered(e -> System.out.print(e + " "));
System.out.println("");

serialStorage
    .stream()
    .forEachOrdered(e -> System.out.print(e + " "));
System.out.println("");

System.out.println("Parallel stream:");
List<Integer> parallelStorage = Collections.synchronizedList(
    new ArrayList<>());
listOfIntegers
    .parallelStream()

    // 不要这样做！它使用了有状态 lambda 表达式。
    .map(e -> { parallelStorage.add(e); return e; })

    .forEachOrdered(e -> System.out.print(e + " "));
System.out.println("");

parallelStorage
    .stream()
    .forEachOrdered(e -> System.out.print(e + " "));
System.out.println("");
```

lambda 表达式 `e -> { parallelStorage.add(e); return e; }` 是有状态 lambda 表达式。每次运行代码时，其结果可能不同。此示例打印以下内容：

```text
Serial stream:
8 7 6 5 4 3 2 1
8 7 6 5 4 3 2 1
Parallel stream:
8 7 6 5 4 3 2 1
1 3 6 2 4 5 8 7
```

操作 `forEachOrdered` 按流指定的顺序处理元素，无论流是串行还是并行执行。但是，当流并行执行时，`map` 操作处理 Java 运行时和编译器指定的流的元素。因此，lambda 表达式 `e -> { parallelStorage.add(e); return e; }` 将元素添加到 `List` `parallelStorage` 的顺序在每次运行代码时可能不同。为了获得确定性和可预测的结果，请确保流操作中的 lambda 表达式参数不是有状态的。

**注意**：此示例调用方法 [`synchronizedList`](https://docs.oracle.com/javase/8/docs/api/java/util/Collections.html#synchronizedList-java.util.List-)，以便 `List` `parallelStorage` 是线程安全的。记住集合不是线程安全的。这意味着多个线程不应同时访问特定集合。假设你在创建 `parallelStorage` 时不调用 `synchronizedList` 方法：

```java
List<Integer> parallelStorage = new ArrayList<>();
```

示例行为不稳定，因为多个线程在没有同步等机制来调度特定线程何时可以访问 `List` 实例的情况下访问和修改 `parallelStorage`。因此，示例可能打印类似以下的输出：

```text
Parallel stream:
8 7 6 5 4 3 2 1
null 3 5 4 7 8 1 2
```
