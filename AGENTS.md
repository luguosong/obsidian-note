# AGENTS.md — Obsidian 笔记仓库协作约定

本仓库是一个 **Obsidian 笔记库**（vault），不是传统软件项目。所有内容为个人知识管理与学习笔记，使用中文撰写。Agent 在此仓库的工作以**创建、编辑、整理笔记**为主。

## 仓库结构

```
obsidian-note/
├── 编程笔记/             # 主要内容（按主题分区）
│   ├── 01-基础知识/
│   ├── 02-前端/
│   ├── 03-Java/
│   ├── 04-专项研究/      # 如 Ai、Claude Code 等专题
│   └── Obsidian/         # Obsidian 使用、插件、markdown 语法
├── 网页裁剪/             # 从网页抓取/裁剪保存的内容
├── 附件/                 # 图片、PDF 等资源
├── 模版/                 # Templater 笔记模版
├── 待办任务/             # 任务清单（见 .gitignore，不纳入版本控制）
├── code/                 # 与笔记配套的代码示例（Java/Node 等）
└── .obsidian/            # Obsidian 配置、插件、主题（纳入版本控制）
```

- **新建笔记默认放入 `编程笔记/` 下对应主题目录**；无法归入既有目录时，与用户确认放置位置。
- 网页内容保存到 `网页裁剪/`；图片等资源保存到 `附件/`。

## 笔记分类机制（核心约定）

`编程笔记/` 采用**「目录即分类」+「文件夹笔记」**的组织方式（配合 Obsidian Notebook Navigator 插件），必须严格遵守：

1. **每个目录都有一个与目录名完全同名的「文件夹笔记」，并且必须放在该目录内部**：路径形如 `目录/目录名.md`。
   - ✅ 正确：`编程笔记/03-Java/01-基础/01-基础.md`（文件夹笔记在 `01-基础` 目录**内部**）
   - ❌ 错误：`编程笔记/03-Java/01-基础.md`（放在父目录里——这会被 Notebook Navigator 视为普通笔记，无法绑定文件夹）
   - 文件夹笔记的文件名 = 目录名，一级标题也与之一致。目录是否使用编号前缀由用户按需决定（如 `01-基础知识` 或 `Ai`），文件夹笔记跟随目录命名即可。
   - 新建目录时**必须同时在该目录内部创建**对应的文件夹笔记。
2. **文件夹笔记之间存在父子关系，通过 `分类` 属性双链指向上一级**：子目录的文件夹笔记，其 `分类` 属性通过双链指向**上一级目录**的文件夹笔记，形成分类树。
   - 例：`编程笔记/03-Java/01-基础/01-基础.md` 的 `分类` 写 `[[03-Java]]`；`编程笔记/03-Java/01-基础/JVM/JVM.md` 的 `分类` 写 `[[01-基础]]`。
   - **例外**：顶层分类目录（如 `01-基础知识`、`03-Java`、`04-专项研究` 等）的文件夹笔记，`分类` 留空，因为它们本身就是最顶层分类。
3. **普通子笔记（非文件夹笔记）的 `分类` 同样指向其所在目录的文件夹笔记**，与文件夹笔记的规则一致。
   - 例：`编程笔记/04-专项研究/Ai/推荐扩展/推荐扩展.md` 属于「推荐扩展」分类，其 frontmatter 写 `分类: - "[[推荐扩展]]"`。
4. 分类可多选：一篇笔记归属多个分类时，`分类` 列出多个 wikilink。

## 笔记编写规范

### Frontmatter（YAML）

每篇笔记必须以 frontmatter 开头。**`编程笔记/` 下的所有笔记统一以 `模版/普通笔记模版.md` 为基础模版**。字段如下：

```yaml
---
分类:                    # wikilink 列表，如  - "[[Ai]]"  - "[[Obsidian]]"
关联笔记:                # 相关笔记的 wikilink 列表（可多值），构建笔记关系网
描述:                    # 一句话说明
排序:                    # 数字，控制在 Notebook Navigator 等插件中的顺序
分组:                    # 可选分组标识
创建时间: 2026年06月24日
修改时间: 2026年06月24日
---
```

本仓库采用**双轴分类设计**，两个字段职责不同，不要混用：

- **分类 = 唯一物理归属（树轴）**。使用带引号的 wikilink 形式：`- "[[分类名]]"`，**指向所在目录的文件夹笔记**（唯一、与目录一致，详见上方「笔记分类机制」）。`分类` 决定笔记在文件树中的位置，永远单一、跟随目录。
- **关联笔记 = 多值关系网（图轴）**。同样是带引号的 wikilink 列表，用于表达笔记与**其它任意节点**的多对多关系——例如一篇笔记在物理上属于 `Spring`，但主题上也涉及 `Java数据库`，就把 `[[Java数据库]]`、`[[事务隔离]]` 等相关节点写进 `关联笔记`，而不必改 `分类`。
  - `分类` 管「这篇笔记放在哪」，`关联笔记` 管「这篇笔记和谁有关系」。跨主题的关联一律走 `关联笔记`，不要为了塞进多个主题而破坏 `分类` 的唯一性。
- **创建时间 / 修改时间**：通过 Templater 插件自动生成（`tp.file.creation_date("YYYY年MM月DD日")` / `tp.file.last_modified_date("YYYY年MM月DD日")`）。格式统一为中文 `YYYY年MM月DD日`（精确到日），手动新建时填当前日期即可，不要回填历史时间。
- 未使用的字段可留空，但**不要删除字段**，保持结构一致。

### 正文

- 首行为一级标题：`# 笔记标题`（与文件名一致）。
- 优先使用 **Obsidian 风格语法**：
  - 内部链接用 **wikilink**：`[[笔记名]]`、`[[笔记名|显示文本]]`、`![[笔记名]]`（嵌入）。
  - 外部链接用标准 Markdown：`[文本](https://...)`。
  - 高亮用 `==文本==`。
- 代码块标注语言：` ```java `、` ```js `、` ```bash ` 等。
- 图片引用：`![[图片名.png]]`，图片文件放在 `附件/`。

## 网页裁剪约定

- **`编程笔记/` 中永远不要放网页裁剪笔记。** 用 `defuddle` skill 抓取的网页内容统一放入 `网页裁剪/` 目录。
- 在 `编程笔记/` 中需要引用网页内容时，用 **wikilink 双链**引用，例如 `[[ClaudeCode在大型代码库中的工作方式：最佳实践与入门]]`。
- 不要把网页正文整段复制进编程笔记，保持「编程笔记写自己的整理与理解，原始内容留在 `网页裁剪/`」的分离。

## `code/` 代码示例目录约定

`code/` 存放与笔记配套的**可运行代码示例**（Java、Node 等小项目）。它与 `编程笔记/` 是「中文笔记 ↔ 英文代码」的配套关系，命名规则截然不同，**必须区分清楚**：

### 1. 结构：与 `编程笔记/` 同构

`code/` 的目录树与 `编程笔记/` **逐级对应**，笔记里的每个分类目录在 `code/` 下都有同位置的英文目录。示例项目放在对应主题目录里。

### 2. 命名：全英文、kebab-case、无编号

`code/` 下的目录和项目**统一使用英文**：

- **去掉编号前缀**（`编程笔记` 的 `01-`、`02-` 等在这里一律不要）。
- **转英文 + kebab-case**（全小写、单词用连字符 `-` 分隔），不要用空格、中文或驼峰。

| `编程笔记/`（中文 + 编号） | `code/`（英文 + kebab + 无编号） |
|---|---|
| `01-基础知识` | `code/basics/` |
| `02-前端` | `code/frontend/` |
| `03-Java` | `code/java/` |
| `03-Java/02-服务端开发/Spring` | `code/java/server/spring/` |
| `04-专项研究` | `code/research/` |

示例项目命名建议带 `-demo` 后缀，如 `code/java-demo`、`code/java/server/spring-ioc-demo`。

### 3. 互动约定

- **笔记引用代码**：在 `编程笔记/` 的笔记里，用代码块路径引用 `code/` 的示例，如 ` ```text \ncode/java-demo\n ``` ` 或在正文中说明路径。
- **代码不要写回笔记目录**：可运行的工程代码只放 `code/`，不要放进 `编程笔记/`（那里只放 `.md` 笔记）。
- **不同步产物**：`code/` 下的编译产物、依赖、IDE 文件已被 `.gitignore` 忽略（`node_modules/`、`target/`、`*.iml` 等），**不要手动提交它们**；只提交源码与必要的配置文件。

## 命名约定


- 笔记文件名用中文，简短有意义，无日期前缀（除非确属日志类）。
- 目录用「`编号-名称`」分区（如 `01-基础知识`、`04-专项研究`），子目录按主题命名。
- **目录深度控制在 3 层以内**（从 `编程笔记/` 算起，如 `编程笔记/03-Java/01-基础/笔记.md`）。更细的主题用 `分类`/`关联笔记` 双链或标签表达，不要再开第 4 层目录——避免路径过长和跨主题笔记被位置锁死。
- 文件名避免特殊字符 `/ \ : * ? " < > |`。

## Git 提交约定

- 自动备份提交信息格式为 `vault backup: YYYY-MM-DD HH:mm:ss`（由 obsidian-git 插件生成，**不要手动模拟该格式**）。
- **手动提交**使用 Conventional Commits 中文描述，例如：
  - `feat: 新增 Claude Code 学习笔记`
  - `chore: 更新插件配置`
  - `docs: 完善模版说明`
- 仅在用户明确要求时执行 `git add` / `commit` / `push`。
- 不要提交 `待办任务/`（已在 `.gitignore`）、`node_modules/`、`*.iml`、IDE 产物等。

## 禁止触碰的内容

- **不要修改 `.obsidian/workspace.json`**（本地窗口布局，频繁变动）。
- 不要擅自删除已有笔记或重构目录结构——涉及整理、迁移时先与用户确认。
- 不要改动 `.obsidian/plugins/` 下插件的内部实现，除非用户明确要求处理插件配置。

## 工作偏好

- 中文沟通，中文撰写笔记与提交信息。
- 涉及创建/编辑 `.md`、`.base`、`.canvas`、Excalidraw 等 Obsidian 文件时，优先调用对应 skill（`obsidian-markdown`、`obsidian-bases`、`obsidian-canvas-creator`、`excalidraw-diagram` 等）。
- 提供网页 URL 时，优先用 `defuddle` skill 提取干净内容再整理入库。
- 不确定分类或位置时，**先问用户**，不要自行假设。
